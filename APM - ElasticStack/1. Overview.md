Dưới đây là **tóm tắt đại cương** về Elastic APM (một phần của Elastic Stack) rồi sau đó mình sẽ đi **vào chi tiết từng thành phần** theo các chủ đề tài liệu mà bạn gửi để bạn hiểu sâu hơn.

---

## 1. Tóm tắt đại cương

* APM (Application Performance Monitoring) là quá trình thu thập, phân tích và phản hồi thông tin về hiệu suất của ứng dụng nhằm nhanh chóng phát hiện và xử lý các vấn đề như độ trễ (latency), lỗi (errors), thông lượng (throughput) và các bottleneck trong hệ thống. ([elastic.co][1])
* Elastic APM là một giải pháp APM tích hợp với Elastic Stack, cho phép bạn instrument ứng dụng (qua agent), gửi dữ liệu về máy chủ APM (APM Server) rồi lưu trữ và phân tích trong Elasticsearch, và hiện thị – điều tra trong Kibana. ([elastic.co][2])
* Với Elastic APM bạn được lợi từ “observability” – không chỉ số liệu hạ tầng (metrics) mà còn tracing (transaction/spans), lỗi (errors), metadata,… giúp hiểu rõ hơn hành vi ứng dụng, nhất là hệ thống phân tán/microservices. ([taloflow.ai][3])
* Luồng hoạt động cơ bản:

  1. Bạn tích hợp APM agent vào ứng dụng (ngôn ngữ Java, Node.js, Python, Go, v.v.).
  2. Agent thu thập thông tin (ví dụ: một HTTP request tới service => transaction; trong đó có các spans như query DB, gọi API ngoài, truy cập cache…).
  3. Agent gửi các sự kiện APM (transaction, span, error, metric) tới APM Server. ([elastic.co][4])
  4. APM Server xử lý, gửi dữ liệu tới Elasticsearch.
  5. Trong Kibana bạn có thể xem dashboard, phân tích, truy vết request xuyên nhiều service (distributed tracing), tìm nguyên nhân gốc rễ (root-cause) khi có sự cố.
* Vì vậy, nếu bạn muốn “làm APM với Elastic Stack”, trước tiên hãy hiểu cái gì sẽ được thu thập, cách tổ chức dữ liệu, cách cài đặt agent/Server, cách cấu hình sampling (chọn mẫu), và cách khai thác dữ liệu (traces, spans, errors, metrics, metadata) để điều tra.

## 2. Đi vào chi tiết

Dưới đây mình lần lượt đi qua các chủ đề bạn gửi: data types, spans, transactions, traces, errors, metrics, metadata, transaction‐sampling. Mình sẽ giải thích khái niệm, cơ chế, và những điểm quan trọng khi triển khai.

### 2.1 Data types (Kiểu dữ liệu trong APM)

![Image](https://www.elastic.co/docs/solutions/images/observability-spans-transactions-and-traces.png)

![Image](https://www.elastic.co/guide/en/observability/8.19/images/apm-geo-ui.png)

![Image](https://www.elastic.co/docs/solutions/images/observability-apm-transaction-sample.png)

![Image](https://www.elastic.co/guide/en/observability/8.19/images/apm-transactions-overview.png)

![Image](https://www.elastic.co/docs/solutions/images/observability-apm-errors-overview.png)

![Image](https://www.elastic.co/docs/solutions/images/observability-apm-app-landing.png)

* Trong Elastic APM, agent thu thập các **event** – mỗi event mô tả một hoạt động hoặc trạng thái. ([elastic.co][5])
* Các kiểu event chính gồm:

  * **Transactions**: đại diện cho một đơn vị công việc lớn trong service (ví dụ: HTTP request, background job). ([elastic.co][6])
  * **Spans**: là các phần nhỏ hơn của transaction – ví dụ: gọi một API ngoài, truy vấn DB, thao tác cache. ([elastic.co][7])
  * **Errors**: các lỗi, exception, sự cố xảy ra trong ứng dụng, được agent tự bắt hoặc bạn có thể instrument bắt. ([elastic.co][2])
  * **Metrics**: các thông số định kỳ về trạng thái service/hệ thống – ví dụ: CPU, GC time, memory usage, số lượng Hàng đợi, v.v. ([elastic.co][8])
* Ngoài ra, mỗi event có thể chứa **metadata** – thông tin bổ sung như labels, user context, environment, version service, host, container, v.v. Mục này làm cho dữ liệu “rich” hơn. ([elastic.co][9])
* Tại sao cần hiểu rõ data types: khi triển khai APM bạn cần biết cái gì được thu thập, cái gì cần instrument thêm, và cách cấu hình sao cho dữ liệu đủ để điều tra – không chỉ “có” mà còn “có đúng”.

### 2.2 Transactions

![Image](https://us1.discourse-cdn.com/elastic/optimized/3X/f/1/f1c234b4e3090634ad791faa7780b9a169e245fb_2_690x338.png)

![Image](https://www.elastic.co/docs/solutions/images/observability-apm-geo-ui.png)

![Image](https://www.elastic.co/docs/solutions/images/observability-spans-transactions-and-traces.png)

![Image](https://us1.discourse-cdn.com/elastic/original/3X/0/d/0d175acb04ac51d1e67db4998d33af2a71a15551.jpeg)

![Image](https://www.elastic.co/docs/solutions/images/observability-apm-transactions-overview.png)

![Image](https://www.elastic.co/docs/solutions/images/observability-apm-transaction-sample.png)

* A transaction là một loại span đặc biệt, có thêm thuộc tính riêng. ([elastic.co][10])
* Ví dụ: khi service bạn nhận một HTTP request, agent sẽ bắt đầu transaction tương ứng với tên service và endpoint đó (ví dụ: `GET /api/orders`). Trong transaction đó có thể có nhiều spans — DB query, cache lookup, external HTTP call…
* Thuộc tính thường bao gồm: `transaction.id`, `trace.id` (nếu phân tán), service name, service version, result (success/failure), duration, outcome, các labels/metadata. ([elastic.co][10])
* Khi nhìn trong Kibana, transaction cho bạn cái nhìn “vĩ mô” cho mỗi request: thời gian total, phân bổ giữa các spans, lỗi nếu có, etc.
* Khi thiết kế: bạn cần đặt tên transaction sao cho rõ, thống nhất (ví dụ theo endpoint, job name), và đảm bảo agent thu đúng mọi request quan trọng (HTTP, background, queue jobs…).

### 2.3 Spans

![Image](https://images.contentstack.io/v3/assets/bltefdd0b53724fa2ce/bltf143ea7f5ff65105/5cb83fb7c5f3a89f491850f3/instrument-apm-go-7.png)

![Image](https://camo.githubusercontent.com/72c22ed508fea362141be565a17f73aa37c15b0569a2a8be46fd95cbad037258/68747470733a2f2f7a7562652e696f2f66696c65732f656c61737469632f66623839353166323663386430623030393033616361336164356465313663302d6b6170747572652d323031392d30322d32382d61742d31342d34382d32332e676966)

![Image](https://www.elastic.co/guide/en/observability/8.19/images/distributed-traces.png)

![Image](https://images.contentstack.io/v3/assets/bltefdd0b53724fa2ce/bltf922d3d931f16598/5c98d5fcd3c73ef55e9ea9c2/blog-opentracing-elastic-apm-6.png)

* Spans mô tả các hoạt động nhỏ hơn bên trong transaction: từ khi span bắt đầu tới khi kết thúc — ví dụ: “SELECT * FROM orders”, “HTTP POST to payment gateway”, “cache get”. ([elastic.co][7])
* Mỗi span có: `span.id`, `parent.id`, `transaction.id`, `trace.id`, start time, duration, name, type/subtype/action (ví dụ, type=`db`, subtype=`mysql`, action=`query`). ([elastic.co][7])
* Quản lý spans giúp bạn hiểu phần nào trong transaction đang tốn thời gian hoặc bị lỗi – rất quan trọng để lý giải bottleneck.
* Một số lưu ý triển khai:

  * Agent mặc định sẽ instrument nhiều thư viện (DB, HTTP client…). Bạn có thể thêm manual spans cho code đặc thù.
  * Cẩn trọng số lượng spans – quá nhiều spans sẽ tạo dữ liệu lớn, có thể ảnh hưởng hiệu suất và chi phí lưu trữ.
  * Cần cấu hình sampling, giới hạn span depth, hoặc tránh instrument quá chi tiết không cần thiết.

### 2.4 Traces

![Image](https://dz2cdn1.dzone.com/storage/temp/15922001-distributedtracing.png)

![Image](https://www.elastic.co/guide/en/observability/8.19/images/distributed-traces.png)

![Image](https://www.elastic.co/guide/en/apm/get-started/6.5/distributed-tracing.jpg)

![Image](https://www.elastic.co/docs/solutions/images/observability-apm-traces.png)

![Image](https://images.contentstack.io/v3/assets/bltefdd0b53724fa2ce/blta65f095d22517ce4/5c98d59849a201165fca1042/blog-opentracing-elastic-apm-3.png)

![Image](https://www.elastic.co/guide/en/apm/get-started/7.10/images/apm-distributed-tracing.png)

* Trace là tập hợp các transactions và spans có chung `trace.id` – thể hiện một luồng request đi qua nhiều service (distributed tracing). ([elastic.co][6])
* Ví dụ: user gửi request tới service A → service A gọi service B → service B gọi DB → service C → trả về. Mỗi service có transaction + spans, tất cả liên kết qua trace.
* Với trace bạn có thể: nhìn thấy đường đi của request, nơi bị chậm, nơi bị lỗi, tương tác giữa các service. Đây là rất cần thiết trong kiến trúc microservices, serverless. ([taloflow.ai][3])
* Triển khai: đảm bảo mọi service có agent, và trace context (header) được truyền qua giữa service (ngôn ngữ/framework cần hỗ trợ). Nếu thiếu instrument hoặc propagate context, trace sẽ bị “ngắt đoạn”.

![Image](https://user-images.githubusercontent.com/5831975/236412102-21571684-253a-4dd9-b3ed-8af3186bc4d1.png)

![Image](https://www.elastic.co/docs/solutions/images/observability-apm-error-group.png)

![Image](https://us1.discourse-cdn.com/elastic/original/3X/7/2/72a55fd58060deb514ca514aef35ff5ad715c97e.png)

![Image](https://opengraph.githubassets.com/181378e9333068b661571050aa12bc02b7a3d4d07e093b0618c641c99e7ddd5f/elastic/apm-agent-nodejs/issues/3960)

![Image](https://user-images.githubusercontent.com/37657851/109029652-5879a700-76c3-11eb-96fa-49df7fb6c835.png)

* Errors là event khi có lỗi (exception, timeouts, HTTP 5xx, v.v) trong quá trình xử lý. Agent tự bắt hoặc bạn có thể instrument bắt. ([elastic.co][2])
* Error event sẽ có: stacktrace, transaction.id + span.id liên quan, user context (nếu cấu hình), labels/metadata.
* Agent và Kibana sẽ nhóm lỗi theo stacktrace và hiển thị số lần xuất hiện, trend tăng/giảm để bạn dễ phát hiện lỗi mới. ([elastic.co][8])
* Khi triển khai:

  * Bật bắt lỗi trong agent.
  * Thiết lập cảnh báo (alerts) nếu lỗi tăng đột biến hoặc lỗi mới xuất hiện.
  * Thiết lập user context/labels để phân biệt ứng dụng/môi trường/version, giúp điều tra nhanh hơn.

### 2.6 Metrics

![Image](https://www.elastic.co/docs/solutions/images/observability-apm-metrics.png)

![Image](https://www.elastic.co/docs/solutions/images/observability-jvm-metrics.png)

![Image](https://learn.microsoft.com/en-us/azure/spring-apps/basic-standard/media/how-to-elastic-apm-java-agent-monitor/elastic-apm-customer-service-jvm-metrics.png)

![Image](https://images.contentstack.io/v3/assets/bltefdd0b53724fa2ce/blt105b9c62fb204735/5f4e4e5d5acde4265bb2dcd3/mysqlservice-metrics.png)

![Image](https://www.elastic.co/guide/en/observability/8.19/images/apm-transactions-overview.png)

![Image](https://www.elastic.co/docs/solutions/images/observability-time-series-expected-bounds-comparison.png)

* Metrics là các số liệu định kỳ về trạng thái hệ thống và ứng dụng – ví dụ: CPU usage, memory, GC time (Java), Go runtime stats, service throughput, latency percentiles. ([elastic.co][8])
* Việc thu metrics giúp bạn có cái nhìn “hạ tầng” hỗ trợ cho việc phân tích transaction/spans/errors – ví dụ: nếu latency tăng thì xem CPU/memory có bị đầy không.
* Trong cấu hình triển khai:

  * Agent sẽ tự thu một số metric mặc định, nhưng bạn có thể instrument thêm custom metrics (ví dụ: queue length, cache hit ratio, custom business metric).
  * Cần cấu hình lưu trữ và retention cho metrics vì có thể rất lớn.
  * Kết hợp metrics với tracing và logs sẽ có cái nhìn toàn diện.

### 2.7 Metadata

![Image](https://images.contentstack.io/v3/assets/bltefdd0b53724fa2ce/blt62e1a3ddd74a6719/63504ad4b3f39b38fccfe7db/blog-elastic-RUMAPM-3.png)

![Image](https://www.elastic.co/docs/reference/apm/images/observability-apm-app-landing.png)

![Image](https://www.elastic.co/guide/en/observability/8.19/images/apm-otel-api-sdk-collector.png)

![Image](https://www.elastic.co/observability-labs/_next/image?q=90\&url=%2Fobservability-labs%2Fassets%2Fimages%2Ffree-open-elastic-apm-observability-deployment%2Fblog-elastic-observability-instrumented-services.png\&w=1920)

![Image](https://opengraph.githubassets.com/9be87d619ef214b4432136525be8ce2f5781caf2bd97716fe165618b57eaad8c/elastic/apm-server/issues/2111)

![Image](https://images.contentstack.io/v3/assets/bltefdd0b53724fa2ce/bltda3d29bf6e005b2a/63504d720528631ea56813e7/blog-elastic-RUMAPM-5.png)

* Metadata là thông tin bổ sung cho các event (transaction, span, error) như: service name, service version, environment (prod/dev), user.id/email, labels, custom context. ([elastic.co][11])
* Ví dụ: bạn có thể gắn label `{ "customer_region": "APAC" }` vào transaction để sau này phân tích theo region. ([elastic.co][9])
* Có hai loại:

  * **Indexed metadata** (ví dụ: labels, user context) — có thể tìm kiếm, group by, aggregate. Cẩn trọng: nếu tạo quá nhiều field thì có thể dẫn tới “mapping explosion”. ([elastic.co][9])
  * **Non-indexed custom context** — chỉ để debug, không để tìm kiếm lớn.
* Triển khai: bạn nên xác định những metadata nào quan trọng với tổ chức (service version, deployment tag, customer id, feature flag) và instrument agent hoặc code để set những thông tin này.

### 2.8 Transaction Sampling (Chọn mẫu giao dịch)

![Image](https://www.elastic.co/docs/solutions/images/observability-dt-sampling-example-1.png)

![Image](https://www.elastic.co/docs/solutions/images/observability-dt-sampling-continuation-strategy-restart_external.png)

![Image](https://www.elastic.co/guide/en/apm/get-started/7.15/images/dt-sampling-example.png)

![Image](https://www.elastic.co/guide/en/observability/8.19/images/apm-transactions-overview.png)

* Vì việc instrument mọi transaction/spans có thể tạo rất nhiều dữ liệu (về volume, lưu trữ, xử lý) — Elastic APM hỗ trợ **sampling** (chỉ thu thập một phần) để cân bằng giữa độ chi tiết và chi phí. ([Last9][12])
* Cơ chế: bạn có thể chỉ thu 100% transaction với spans ban đầu, rồi chỉ lưu lại 1% cá nhân sau khi đã instrument (tùy cấu hình). Hoặc bạn có thể loại trừ các transaction nhỏ, ít quan trọng.
* Khi cấu hình sampling bạn cần cân nhắc:

  * Transaction quan trọng (ví dụ, checkout, payment) nên giữ 100%.
  * Transaction ít giá trị phân tích (ví dụ internal health check) có thể sampling thấp.
  * Khi giảm sampling, có thể mất visibility về một số trường hợp hiếm hoặc lỗi. Vì vậy cần cân nhắc trade-off.
  * Kiểm soát spans depth và size dữ liệu – nếu transaction có rất nhiều spans mà bạn không cần hết, bạn có thể cấu hình “span_min_duration” hoặc “span_exclusion” để tránh instrument quá chi tiết.

## 3. Kết lại & liên hệ với môi trường của bạn

Với mục tiêu của bạn — bạn muốn “làm APM với Elastic Stack” — thì từ việc hiểu đại cương và chi tiết như trên, bạn có thể tiến tới các bước triển khai thực tế như:

1. Chọn agent phù hợp với ngôn ngữ/ứng dụng hiện tại của bạn (ví dụ Java, Go, Node.js).
2. Cấu hình agent: service name, version, environment, metadata custom, error catching.
3. Triển khai APM Server (hoặc sử dụng Elastic Cloud) và cấu hình kết nối tới Elasticsearch.
4. Kiểm soát sampling để đảm bảo dữ liệu không quá lớn nhưng vẫn đủ thông tin.
5. Thiết lập dashboards/alerts trong Kibana để theo dõi transaction latency, error rate, traces đặc biệt, bottlenecks.
6. Trong môi trường của bạn (như hệ thống Kubernetes/microservices) — đảm bảo context tracing được propagate giữa service, agent được instrument đúng, và bạn có metadata để phân biệt service version, namespace, deployment, customer.
7. Xây dựng quy trình vận hành: khi latency tăng, trace xem spans; khi error tăng, drill-down vào stacktrace; kết hợp metrics → tracing → logs để điều tra root-cause.
